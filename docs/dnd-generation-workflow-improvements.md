# D&D Generation Workflow Improvements

> Created: 2025-02-13
> Status: In Progress
> Last Updated: 2026-03-02

# Overview

This document tracks all identified improvements to the D&D content generation workflows. The NPC pipeline is the most mature; other content types need similar treatment.

---

## Gemini Provider-First Stage Automation (NEW)

### Goal
When a Gemini provider is configured, run stages automatically with minimal user interaction while keeping Manual Copy/Paste as a first-class peer mode (not a fallback). Manual mode must remain end-to-end usable for users who prefer other AIs or when provider limits are reached.

### Primary guardrails
- Deterministic apply: never apply the same provider response twice.
- No silent fixes: illegal states must surface as explicit errors.
- Schema purity: validate patch against the stage schema before applying.
- Scope safety: never allow a patch to mutate outside the stage’s permitted fields.
- Manual is always available and never loses progress.

### Behavior

#### Mode selection
Two modes exist and are always selectable:
- Provider Mode (Gemini)
- Manual Mode (Copy/Paste)

Provider Mode can be defaulted per project/user, but the user can switch at any time without losing state.

#### Provider Mode: stage lifecycle
On stage open, if provider configured and stage not complete:
1) Auto-send stage prompt to Gemini (exactly once per stage-open event)
2) Await response
3) Extract JSON patch
4) Validate patch (schema + scope)
5) Apply patch (deep-merge rules)
6) Mark stage complete
7) Auto-advance

If any error (parse, validation, scope, merge, provider/rate-limit):
- Show clear messaging with Retry
- Manual Mode remains visible and selectable without resetting stageResults

#### Manual Mode: stage lifecycle
- User manually runs any AI and pastes response
- System performs the same JSON extraction + validation + apply + complete + advance steps
- Manual Mode is not treated as “fallback”; it is a supported primary workflow

#### Critical gotcha prevention
- Auto-run must not re-trigger on re-render, stage panel open/close, or unrelated state updates
- Auto-run must be gated by a per-stage run token (see “Idempotency & Concurrency”)

### Backend

#### Endpoint
`POST /api/ai/gemini/generate`

#### Request
```
{
  projectId,
  stageId,
  stageRunId,
  prompt,
  schemaVersion,
  responseFormat: "json_patch_preferred",
  clientContext: {
    appVersion,
    stageKey,
    generatorType,
    userSelectedMode
  }
}
```

Notes:
- `stageRunId` is required for idempotency and replay protection.

#### Success response
```
{
  ok: true,
  provider: "gemini",
  model,
  requestId,
  stageRunId,
  rawText,
  jsonPatch?,
  parse: { foundJsonBlock, parseWarnings: [] },
  usage: { inputTokens, outputTokens },
  safety: {
    patchSizeBytes,
    appliedPathsCandidateCount
  }
}
```

#### Failure response
```
{
  ok: false,
  requestId,
  stageRunId,
  error: {
    type: RATE_LIMIT | PROVIDER_ERROR | INVALID_RESPONSE | TIMEOUT | SCHEMA_MISMATCH | FORBIDDEN_PATH | PAYLOAD_TOO_LARGE | BUDGET_EXCEEDED | ABORTED,
    message,
    retryable,
    retryAfterMs?
  }
}
```

#### Secrets and key handling
- Server holds the Gemini key; browser never sees it.
- Keys stored in server secret manager; support rotation with `activeKeyId`.
- Never log keys, never echo keys to clients.

#### Rate limiting and quotas
- Per-user and per-project rate limiting (token bucket).
- Server-side concurrency cap per user/project (e.g., 1 in-flight stage call at a time by default; configurable).
- Retries with exponential backoff on retryable errors (cap: 2 retries).
- Circuit breaker on repeated 429s (cooldown: e.g., 60s).
- Return actionable guidance: “Rate limit exceeded; wait X seconds or switch to Manual Mode.”

#### Budget guardrails (strongly recommended)
- Add project-level and account-level monthly budget ceiling (soft stop + hard stop).
- If budget exceeded:
  - Provider calls blocked with `BUDGET_EXCEEDED`
  - Manual Mode remains available

#### Logging and privacy
- Production logs must not include full prompt/response bodies.
- Log metadata only: requestId, stageRunId, projectId, stageId, model, durationMs, token usage, error type.
- Optional secure audit store (opt-in) can store redacted prompt/response for debugging with retention limits.

### Idempotency & Concurrency (NEW – critical guardrails)

#### Why
Retries and UI re-renders can cause duplicate applies (data corruption or repeated stage completion). This must be prevented.

#### Rules
- Every stage execution has a unique `stageRunId` generated by the frontend when the stage opens (UUID).
- Backend must treat `(projectId + stageId + stageRunId)` as an idempotency key: if the same `stageRunId` is received again, return the previously computed response (or a stable “already completed” response).
- Frontend must refuse to apply a response if:
  - `stageRunId` mismatch, or
  - stage already marked complete, or
  - response already applied (store appliedRequestId per stage)

#### Cancellation
- Cancel must abort the in-flight request (AbortController on client; server should honor if possible).
- Cancel transitions stage state to idle (or error with ABORTED) without applying anything.
- Cancel must not block subsequent retries.

### JSON Extraction & Validation

#### Extraction (strict)
- Prefer fenced ```json block
- Else first balanced top-level `{ ... }` region
- Strict JSON parse (no trailing commas, no comments)
- On parse failure: return `INVALID_RESPONSE`, include location info (line/col if feasible), and a short “how to fix” message for manual users

#### Validation stages
- Schema validation: validate `jsonPatch` against the stage schema
- Scope validation (NEW): validate that the patch only touches allowed fields for this stage
- Size validation (NEW): reject excessive payloads (e.g., > 200KB patch) with `PAYLOAD_TOO_LARGE`

#### No heuristic inference
- Do not infer mechanics from prose.
- Do not auto-inject missing required fields. Missing required fields must be a validation error.

#### SchemaVersion drift handling (NEW)
- If `schemaVersion` in request is not the current `schemaVersion` for that stage: backend returns `SCHEMA_MISMATCH` with expected version; frontend prompts user to refresh/re-run stage with the updated schema.

### Patch Contract and Safety (NEW – critical)

#### Patch shape
- `jsonPatch` represents only fields to merge, not full object dumps.
- Must be an object at the root.

#### Forbidden patch behaviors
- Overwrite root `stageResults` wholesale
- Modify other stages’ keys
- Modify system fields (projectId, timestamps, ownership, etc.)
- Write to reserved meta paths
- Any of the above must fail with `FORBIDDEN_PATH`.

#### Allowed path list
- Each stage defines an explicit allowlist of JSON pointer paths or top-level keys it may modify.
- Scope validation uses this allowlist.

#### Optional patch metadata (recommended)
```
{
  "_meta": {
    "stageId": "...",
    "schemaVersion": "...",
    "notes": "..."
  },
  "...actual fields..."
}
```
`_meta` is ignored for merge except for diagnostics.

### Merge Rules
- Deep merge semantics
  - Objects: deep merge recursively
  - Arrays: replace-by-default
- Optional future directive: `{ "someArray": { "$append": [ ... ] } }` (not implemented now)
- Never clobber sibling stage results; merge is restricted to the stage’s namespace. Any non-stage keys in `jsonPatch` are rejected.
- Deterministic merge output: same inputs → same outputs.

### Frontend Stage Runner

#### State machine per stage
`idle → sending → awaiting_ai → parsing → validating → applying → complete | error`

#### Auto-run rules
- Auto-run once per stageOpen event only if:
  - provider configured
  - mode is Provider Mode
  - stage not complete
  - `stageRunId` is new
- Provide “Resend” explicitly; do not auto-resend on error.

#### UI requirements
- Show status for each stage (sending/awaiting/parsing/validating/applying).
- Cancel button only visible while sending/awaiting.
- Retry visible on error.
- Manual Mode toggle always visible.
- On provider error: show manual instructions without losing state.

#### Advance rules
- Auto-advance only after: schema validation passes, scope validation passes, merge apply succeeds, stage complete flag persisted.

#### Persistence gotcha guardrail (NEW)
- Ensure stage completion and `stageResults` update are saved atomically: either both persist, or neither persists. No “completed stage with missing applied patch.”

### Manual Copy/Paste (First-Class Mode)
- Always available, visible regardless of provider configuration.
- Supports end-to-end manual completion of all stages.
- Uses identical extraction + validation + merge pipeline.
- Shows the same field-level errors.

#### Friction reducers (optional)
- “Copy Prompt” button
- “Paste From Clipboard” button (requires user gesture)
- “Validate Only” button (lets users see errors before applying)

### Acceptance Criteria (expanded)

#### Happy path
- Provider configured + Provider Mode selected
- Stage auto-sends exactly once on open
- Valid JSON patch returned
- Schema + scope validation pass
- Deep merge applies deterministically
- Stage completes and persists
- Auto-advances to next stage

#### Error handling
- Invalid JSON → error shown with parse location; no advance; Manual Mode available
- Schema violation → field-level errors; no advance; Manual Mode available
- Forbidden paths → `FORBIDDEN_PATH`; no apply; no advance
- Provider 429/503 → backoff; actionable error; no advance; Manual Mode available
- Missing/invalid key → clear error; Manual Mode available
- Cancel mid-flight → request aborted; no apply; stage remains incomplete

#### Integrity and idempotency
- Retrying the same `stageRunId` does not duplicate apply
- Rapid stage switching does not trigger multiple in-flight calls
- `stageResults` integrity preserved (no unrelated overwrites)
- Atomic persistence: no “complete without patch” state

#### Security
- API key never present in frontend bundle
- Per-user/project rate limiting enforced server-side
- Prompts/responses not logged in production

## Architecture Patterns (Reference)

All new pipelines should follow the NPC pattern:

1. **Schema** — `schema/{type}/v1.1-client.json` + `v1.1-server.json`
2. **Schema Extractors** — `client/src/utils/{type}SchemaExtractor.ts` with per-stage extraction functions
3. **Creator Stages** — `client/src/config/{type}CreatorStages.ts` with focused sub-stages
4. **Section Chunks** — `client/src/config/{type}SectionChunks.ts` for section-based chunking (optional)
5. **Stage Router** — `client/src/config/{type}StageRouter.ts` for smart conditional routing (optional)
6. **Stage Merger** — `client/src/utils/{type}StageMerger.ts` for multi-stage output merging
7. **Server Normalization** — `src/server/services/generatedContentMapper.ts` normalize + format functions
8. **Server Validation** — `src/server/validation/{type}Validator.ts` using server schema
9. **ManualGenerator Integration** — `getStages()` branch + stage key list in `handleAcceptWithIssues`

## Coding Standards

- Follow existing patterns in NPC pipeline files
- Use TypeScript strict mode compatible types
- Include JSDoc comments on exported functions
- Use `ensureString`, `ensureArray`, `ensureObject` helpers from generatedContentMapper
- Schema extractors return `SchemaObject` type
- Stage prompts must include `⚠️ CRITICAL OUTPUT REQUIREMENT` JSON-only instruction
- All stage `buildUserPrompt` must include `previous_decisions` and `canon_reference` handling
- Section chunks must define `outputFields` for merger field provenance
